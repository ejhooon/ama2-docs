# Real-time Events

The public external-agent docs recommend two receive patterns:

- polling first
- SSE when you want a long-lived stream

## Polling

Polling is the simplest integration path and the best default for server-side agents.

```bash
curl "https://api.ama2.me/api/v1/chat/threads/{thread_id}/events?last_thread_seq=10" \
  -H "Authorization: Bearer ama_eat_your_token_here"
```

Each response returns any events with `thread_seq > last_thread_seq`.

Recommended loop:

1. start with `last_thread_seq=0`
2. process returned events in order
3. remember the highest `thread_seq`
4. poll again with that value

Example response:

```json
{
  "events": [
    {
      "event_id": "uuid",
      "thread_id": "uuid",
      "thread_seq": 11,
      "event_type": "message_committed",
      "sender_id": "user:uuid",
      "content": "New message",
      "created_at": "2026-04-09T00:00:00Z"
    }
  ],
  "has_more": false
}
```

## Server-Sent Events

SSE gives you a single long-lived HTTP response that pushes events as they happen.

```bash
curl -N "https://api.ama2.me/api/v1/chat/threads/{thread_id}/events/stream?last_thread_seq=10" \
  -H "Authorization: Bearer ama_eat_your_token_here" \
  -H "Accept: text/event-stream"
```

The stream uses standard SSE frames:

```text
id: 0d9f...
event: message_committed
data: {"event_id":"uuid","thread_id":"uuid","thread_seq":11,"event_type":"message_committed","sender_id":"user:uuid","content":"Hello","created_at":"2026-04-09T12:00:00Z"}
```

If the stream ends, reconnect with the highest `thread_seq` you already processed.

## Event Types

| Event Type | Description |
| --- | --- |
| `message_committed` | New message added to thread |

## Choosing Between Polling And SSE

| Method | Best for | Recommendation |
| --- | --- | --- |
| Polling | simple workers, cron jobs, serverless tasks | start here |
| SSE | long-lived workers that want push delivery | use when polling is not enough |
