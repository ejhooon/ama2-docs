# Threads

A thread is the persistent conversation container your external agent reads from and writes to.

These docs focus on the private-thread runtime used by external agent integrations.

## What A Thread Gives You

A thread gives you:

- a stable `thread_id`
- a participant list
- ordered message events via `thread_seq`
- a place to send new agent messages
- a replay-safe event cursor for polling and SSE

## Private Thread Creation

Use `POST /api/v1/chat/threads` to create or converge a private thread.

The request body uses:

```json
{
  "participant_user_ids": ["user-uuid-1", "user-uuid-2"]
}
```

If the same active participant set already has a private thread, AMA2 returns that thread instead of creating a duplicate.

## Common Thread Fields

| Field | Type | Description |
| --- | --- | --- |
| `thread_id` | UUID | Unique identifier |
| `participants` | array | List of participants with roles |
| `is_archived` | boolean | Whether the thread is archived |
| `last_message_at` | timestamp | Most recent message time |
| `last_message_preview` | string | Summary of the latest visible message |
| `participation_mode` | `"open"` \| `"invocation_only"` | Agent response behavior |

## Participants

Participants are represented by `sender_id` values such as:

- `user:<uuid>`
- `agent:<uuid>`

An external agent must be a thread participant before it can send or read thread events.

## Participation Mode

`participation_mode` controls whether the enrolled agent can answer freely or only when mentioned.

| Mode | Meaning |
| --- | --- |
| `open` | the agent may send a reply normally |
| `invocation_only` | the agent may send only after it was mentioned in the latest human message |

## Event Ordering

Every committed event in a thread carries a monotonically increasing `thread_seq`.

Use `thread_seq` to:

- resume polling without reading the full history again
- reconnect an SSE consumer after interruption
- attach freshness metadata when your agent sends a reply
