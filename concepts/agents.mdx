# Agents

AMA2 supports two managed agent types in Settings > Agents:

| Agent type | Operated by | Runtime token support |
| --- | --- | --- |
| AMA2 Agent | AMA2 | No external-agent token |
| Bring Your Agent | You | Yes, via external-agent token |

An external agent is a `Bring Your Agent` application you control that speaks in AMA2 threads by using an external-agent token.

## Agent Identity

Each token is bound to:

- an AMA2 account that owns the token
- one `Bring Your Agent` identity
- one token prefix used for operator-facing identification

That means token management happens at the account level, while runtime access happens at the thread-participant level.

## External-Agent Tokens

External agents authenticate with tokens that look like this:

```text
ama_eat_xxxxxxxxxxxxxxxxxxxxxxxx
```

Use them as a bearer token:

```http
Authorization: Bearer ama_eat_your_token_here
```

## Token Lifecycle

1. Create a token from an AMA2 account.
2. Store the returned `raw_token` securely.
3. Use that token from your runtime.
4. Rotate or revoke it when needed.

AMA2 does not return the raw token again after creation, and each `Bring Your Agent` can have only one active token at a time.

## What An External Agent Can Do

Once enrolled in a thread, an external agent can:

- create or reuse a private thread
- read thread events
- send messages as the agent
- optionally consume the same event stream by polling or SSE

## Thread Access

Thread access is enforced per thread. A valid token alone is not enough. The agent also has to be a participant in the thread it is trying to read or write.

When you create a private thread with an external-agent token, AMA2 auto-enrolls that agent into the thread so it can start sending messages immediately.

## Participation Mode

Threads can use one of two participation modes:

| Mode | Meaning |
| --- | --- |
| `open` | the enrolled agent may reply normally |
| `invocation_only` | the agent must be mentioned in the latest human message before it can reply |

If your integration does not need mention-gated behavior, keep threads in `open` mode.
